# 运行时配置

本文档描述已落地的运行时数据库与模型路由配置能力：前端可创建短生命周期运行配置，覆盖 `workflow.yaml` 的默认数据库连接和 `light`/`strong` 模型路由，查询、Schema 读取和编辑 SQL 执行都复用同一配置会话。判断当前行为时，以源码、测试和本文档为准。

## 能力概述

运行时配置不改变 `WorkflowEngine`、`NodeFactory`、`NodeRegistry` 的职责，只通过请求级依赖注入生效：

1. 前端创建短生命周期运行配置，获得 `runtime_config_id`。
2. 数据库可选 `workflow.yaml` 预设连接，或提交自定义 SQLite/PostgreSQL/MySQL 连接。
3. `light` 和 `strong` 两档模型必须同时配置，可用预设或自定义 OpenAI-compatible / Mock provider；两档可来自不同 provider、不同 base URL、不同 API Key。
4. 创建配置时统一测试数据库连接和模型连通性，任一失败则不保存。
5. 后续 `/api/v1/query`、`/api/v1/schema`、`/api/v1/sql/execute` 可携带 `runtime_config_id` 使用该配置。
6. 不传 `runtime_config_id` 时继续使用默认 `workflow.yaml` 配置。
7. 用户提交的数据库密码和模型 API Key 只保存在后端内存运行时会话中，不落盘、不回传前端。

## 接口

### 获取系统选项

```http
GET /api/v1/runtime/options
```

返回前端配置页可选的预置数据库和预置模型。`sqlite_file_*` 来自 `database.sqlite_discovery` 对 `data/sqlite/` 的自动扫描；`metadata.db` 默认排除。响应只返回预设摘要，不返回真实 API Key、数据库密码或完整数据库 URL。

```json
{
  "database_presets": [
    {"id": "demo_sqlite", "driver": "sqlite", "display_name": "demo_sqlite", "target_dialect": "sqlite", "read_only": true},
    {"id": "sqlite_file_northwind", "driver": "sqlite", "display_name": "northwind.db", "target_dialect": "sqlite", "read_only": true}
  ],
  "model_presets": {
    "light": [{"id": "light", "provider": "openai_compatible", "model": "deepseek-v4-flash", "display_name": "openai_compatible/deepseek-v4-flash", "requires_secret": true}],
    "strong": [{"id": "strong", "provider": "openai_compatible", "model": "deepseek-v4-pro", "display_name": "openai_compatible/deepseek-v4-pro", "requires_secret": true}]
  }
}
```

### 创建运行时配置

```http
POST /api/v1/runtime/configs
```

当前没有独立的数据库测试或模型测试接口，连通性检查发生在创建配置时：`TextToSQLApiService.create_runtime_config` 先调用 `RuntimeConfigTester.test_database` 和 `RuntimeConfigTester.test_model`，任一失败返回脱敏错误响应，不保存该配置。

预置数据库请求示例：

```json
{
  "database": {"mode": "preset", "preset_id": "demo_sqlite"},
  "models": {
    "light": {"mode": "preset", "preset_id": "light"},
    "strong": {"mode": "preset", "preset_id": "strong"}
  }
}
```

自定义数据库 + 自定义模型请求示例：

```json
{
  "database": {
    "mode": "custom",
    "config": {
      "driver": "postgresql", "host": "localhost", "port": 5432,
      "database_name": "analytics", "username": "readonly_user",
      "password": "runtime only", "target_dialect": "postgres", "display_name": "analytics"
    }
  },
  "models": {
    "light": {"mode": "preset", "preset_id": "light"},
    "strong": {
      "mode": "custom", "provider": "openai_compatible", "model": "gpt-4.1",
      "base_url": "https://api.openai.com/v1/chat/completions", "api_key": "runtime only"
    }
  },
  "ttl_seconds": 7200
}
```

成功响应（不含密码和 API Key）：

```json
{
  "runtime_config_id": "rt_abc123",
  "expires_at": "2026-06-23T12:30:00Z",
  "database": {"display_name": "analytics", "driver": "postgresql", "target_dialect": "postgres", "table_count": 12, "column_count": 48, "tables": ["customers", "orders"]},
  "models": {
    "light": {"provider": "openai_compatible", "model": "deepseek-v4-flash"},
    "strong": {"provider": "openai_compatible", "model": "gpt-4.1"}
  }
}
```

### 业务接口使用运行时配置

```http
POST /api/v1/query          # 请求体增加可选 runtime_config_id
GET /api/v1/schema?runtime_config_id=rt_abc123
POST /api/v1/sql/execute    # 请求体增加可选 runtime_config_id
```

## 后端模块

```text
src/text_to_sql_demo/runtime/
├── __init__.py
├── models.py        # 运行时配置请求、响应和内部模型
├── store.py         # 内存保存 RuntimeConfig，支持 TTL 和过期清理
├── resolver.py      # 把 preset/custom 配置解析为数据库 URL、方言、LLM client、模型 profiles
├── tester.py        # 创建配置时测试数据库连接和模型连通性，返回脱敏摘要
└── exceptions.py    # 运行时配置专用异常
```

接入点：

- `main.py`：注册 `/api/v1/runtime/options`、`/api/v1/runtime/configs`，`GET /api/v1/schema` 接收可选 `runtime_config_id`。
- `api/models.py`：`QueryRequest` 和 `ExecuteSQLRequest` 增加 `runtime_config_id`。
- `api/service.py`：解析运行时配置，读取对应 schema，向 workflow 注入数据库 URL、LLM client 和模型 profiles。
- `llm/routing.py`：`RoutingLLMClient` 按 alias 分发到不同 provider client。

## 内部模型

```python
class RuntimeConfig(BaseModel):
    id: str
    expires_at: datetime
    database: RuntimeDatabaseConfig
    models: RuntimeModelRoutingConfig
    display: RuntimeConfigDisplay

class RuntimeDatabaseConfig(BaseModel):
    driver: Literal["sqlite", "postgresql", "mysql"]
    database_url: SecretStr
    target_dialect: Literal["sqlite", "postgres", "mysql"]
    display_name: str

class RuntimeModelRoutingConfig(BaseModel):
    light: RuntimeModelConfig
    strong: RuntimeModelConfig

class RuntimeModelConfig(BaseModel):
    provider: str
    model: str
    base_url: str | None = None
    api_key: SecretStr | None = None
    api_key_env: str | None = None
    temperature: float = 0.0
    max_tokens: int | None = None
```

`RuntimeConfigStore` 使用内存 dict，`ttl_seconds` 默认 7200 秒。服务重启后临时配置失效。

## LLM 路由

`light` 与 `strong` 可使用不同 provider 或 key，因此运行时配置通过 `RoutingLLMClient` 按 alias 分发：

```python
class RoutingLLMClient:
    clients_by_alias: dict[str, LLMClient]
    def complete(..., model_alias: str, ...) -> LLMResponse:
        return self.clients_by_alias[model_alias].complete(...)
```

`GenSQLAgenticNode` 和 `FixSQLNode` 不需要改调用方式，仍传入 `model_alias`，底层由 `RoutingLLMClient` 转发到对应 provider client。

## SQL 方言与 driver 映射

| driver | target_dialect | execution_dialect |
| --- | --- | --- |
| `sqlite` | `sqlite` | `sqlite` |
| `postgresql` | `postgres` | `postgres` |
| `mysql` | `mysql` | `mysql` |

使用 `runtime_config_id` 时，后端按本次配置生成请求级 workflow config copy（不写回 `workflow.yaml`）：

1. `state.data["target_dialect"]` 使用 runtime dialect。
2. `sql_generation.target_dialect` 使用 runtime dialect。
3. `sql_validation.target_dialect` 和 `render_dialect` 使用 runtime dialect。
4. `sql_execution.execution_dialect` 使用 runtime dialect。

## 前端交互

前端在 `frontend/src/App.tsx` 中实现运行配置侧边面板，不把配置表单堆进主查询区。

页面结构：

```text
顶部栏
├── 当前运行配置摘要（数据库 / 轻量模型 light / 强模型 strong）
└── 配置按钮

运行配置面板
├── 数据库连接（使用系统预置连接 / 自定义连接）
├── 模型路由（轻量模型 light / 强模型 strong）
├── 保存时统一连通性检查
└── 保存并启用
```

前端保存内容：`runtime_config_id`、`expires_at`、脱敏后的数据库和模型摘要（保存在 React state）。
前端不保存：数据库密码、API Key、完整数据库 URL（不写入 `sessionStorage`）。

后端返回过期或不存在时，前端展示错误提示，用户可清除运行配置后重新创建。

## 错误码

| 错误码 | 含义 |
| --- | --- |
| `runtime_config_not_found` | 找不到运行时配置 |
| `runtime_config_expired` | 运行时配置已过期 |
| `runtime_config_invalid` | 运行时配置结构无效 |
| `runtime_preset_not_found` | 数据库或模型预设不存在 |
| `runtime_connection_failed` | 创建配置时数据库或模型连通性检查失败 |
| `runtime_secret_missing` | 缺少 API Key 或数据库密码 |
| `runtime_provider_unsupported` | provider 不支持 |
| `validation_error` | 请求体未通过 Pydantic/FastAPI 校验，例如缺少 `light` 或 `strong` |

错误响应使用现有统一结构，`details` 中不能包含密码、API Key、Authorization、完整数据库 URL、完整 prompt、完整 SQL 或完整结果集。

## 安全约束

允许记录：`request_id`、`runtime_config_id`、`driver`、`provider`、`model_alias`、`model_name`、`connection_display_name`、`error_type`、异常源文件和行号。

禁止记录：`api_key`、`password`、`Authorization`、完整数据库 URL、完整 prompt、完整 SQL、完整结果集。数据库 URL 只能记录脱敏版本或 hash。SQL 默认只记录长度和 hash；debug 明确开启时才允许有限 preview。

## 边界

- 运行时配置只保存在后端内存，服务重启后失效，不做跨实例共享。
- 不实现用户登录、认证、多租户或权限系统。
- 不做 API Key、数据库密码的持久化密钥管理。
- SQL 执行仍以 demo 只读校验为主，不等同于生产级数据库权限隔离。
- 不引入 Redis、向量数据库或其他分布式基础设施。
