# 节点扩展开发

本文档面向二次开发者，说明如何新增一个工作流节点。项目核心卖点是"注册式节点体系，新增节点不需要改动工作流引擎或工厂"，本文档把这条链路从源码讲清楚。

## 核心机制

节点扩展依赖三个组件，都在 `src/text_to_sql_demo/workflow/` 下：

```text
workflow/
├── node.py          # BaseNode 抽象基类 + NodeResult
├── registry.py      # NodeRegistry 注册表 + @register_node 装饰器
├── factory.py       # NodeFactory 按 type 创建节点
├── dependencies.py  # NodeDependencies 依赖容器
├── state.py         # WorkflowState 类型化状态
└── engine.py        # WorkflowEngine 执行器（不感知具体节点类）
```

**关键设计**：`WorkflowEngine` 和 `NodeFactory` 都**不导入具体节点类**，只通过 `NodeRegistry` 按 `type` 字符串解析。这套约束由 `tests/unit/workflow/test_architecture_constraints.py` 做 AST 检查保护。

## BaseNode 接口

`BaseNode`（`src/text_to_sql_demo/workflow/node.py`）定义四个方法：

```python
class BaseNode(ABC):
    def __init__(self, *, name, config, dependencies, node_type): ...

    def before(self, state: WorkflowState) -> None:
        """run 前立即调用的生命周期钩子，默认空实现。"""

    @abstractmethod
    def run(self, state: WorkflowState) -> NodeResult:
        """执行节点并返回结构化结果，子类必须实现。"""

    def after(self, state: WorkflowState, result: NodeResult) -> None:
        """run 成功后调用的生命周期钩子，默认空实现。"""

    def error(self, state: WorkflowState, error: Exception) -> None:
        """run 抛出异常时调用的生命周期钩子，默认空实现。"""
```

构造参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `name` | `str` | 节点实例名，来自 `workflow.yaml` 的 `nodes` 键名 |
| `config` | `Mapping[str, Any]` | 节点配置，来自 `workflow.yaml` 的 `nodes.<name>` 下除 `type` 外的字段 |
| `dependencies` | `NodeDependencies` | 共享依赖容器，由 `TextToSQLApiService` 注入 |
| `node_type` | `str` | 节点类型名，来自 `workflow.yaml` 的 `nodes.<name>.type` |

## NodeResult 结构

`run` 方法返回 `NodeResult`（`src/text_to_sql_demo/workflow/node.py`）：

```python
class NodeResult(BaseModel):
    outcome: str = "success"                      # 决定下一节点的路由键
    state_patch: dict[str, Any] = {}              # 合并到 WorkflowState 的 patch
    output: dict[str, Any] = {}                   # 节点输出摘要，进入 Trace
    terminate: bool = False                       # 是否终止工作流
    termination_reason: str | None = None         # 终止原因
```

**`outcome` 是路由的核心**：`WorkflowEngine._resolve_next_node` 用当前节点的 `outcome` 在 `workflow.yaml` 的 `edges` 里找下一节点。

**`state_patch` 合并规则**：deep merge 到 `WorkflowState`。约定俗成的写法是 `{"data": {"your_key": value}}`，把业务中间结果放在 `state.data` 下。

## WorkflowState 结构

`WorkflowState`（`src/text_to_sql_demo/workflow/state.py`）是 typed Pydantic model，业务中间结果集中在 `state.data`（`dict[str, Any]`）。常用键：

| 键 | 写入节点 | 说明 |
| --- | --- | --- |
| `task` | BeginNode | 任务元信息 |
| `intent` | SelectionNode | 意图分类 |
| `schema` | API service | 完整 schema |
| `schema_linking` | SchemaLinkingNode | linked schema 子集 |
| `rag_context` | ContextRetrievalNode | 知识库 Top-K |
| `retrieved_examples` | ExampleRetrievalNode | SQL 示例 Top-K |
| `generated_sql` | GenSQLAgenticNode | 生成的 SQL |
| `validated_sql` | ValidateSQLNode | 校验通过的 SQL |
| `execution_result` | ExecuteSQLNode | 执行结果 |
| `last_error` | ValidateSQLNode/ExecuteSQLNode | 结构化 SQLError |
| `reflection_decision` | ReflectionDecisionNode | 反思策略 |
| `repair_history` | FixSQLNode | 修复历史 |
| `attempt_count` | FixSQLNode/ReasoningRewriteNode | 修复尝试次数 |
| `final_status` | FinalizeNode | 最终状态 |

完整字段见 [整体架构](整体架构.md) 的"状态与 Trace"小节。

## 注册节点

用 `@register_node` 装饰器（`src/text_to_sql_demo/workflow/registry.py`）注册：

```python
from text_to_sql_demo.workflow.node import BaseNode, NodeResult
from text_to_sql_demo.workflow.registry import register_node
from text_to_sql_demo.workflow.state import WorkflowState


@register_node("my_custom_node")
class MyCustomNode(BaseNode):
    """我的自定义节点。"""

    def run(self, state: WorkflowState) -> NodeResult:
        # 读 state
        question = state.user_question
        linked = state.data.get("schema_linking")

        # 读 config（来自 workflow.yaml 的 nodes.<name> 下除 type 外的字段）
        my_option = self.config.get("my_option", "default")

        # 读 dependencies（共享依赖）
        llm_client = self.dependencies.get("llm_client")

        # 业务逻辑...

        return NodeResult(
            outcome="success",
            state_patch={"data": {"my_result": "value"}},
            output={"my_result": "value"},
        )
```

**注册触发时机**：节点类必须在 `src/text_to_sql_demo/nodes/__init__.py` 中被 import，才会触发 `@register_node` 装饰器执行。新增节点后，在 `__init__.py` 加一行 import：

```python
from text_to_sql_demo.nodes.my_custom import MyCustomNode
```

## NodeDependencies 依赖注入

`NodeDependencies`（`src/text_to_sql_demo/workflow/dependencies.py`）是 frozen dataclass，统一注入共享依赖：

```python
@dataclass(frozen=True, slots=True)
class NodeDependencies:
    values: Mapping[str, Any]
    def get(self, name: str, default: Any = None) -> Any: ...
```

`TextToSQLApiService` 在创建 `WorkflowEngine` 时注入以下依赖：

| 依赖键 | 类型 | 说明 |
| --- | --- | --- |
| `database_url` | `str` | 业务目标库连接 URL |
| `llm_client` | `LLMClient` | LLM 客户端（Mock 或 OpenAI-compatible 或 RoutingLLMClient） |
| `model_profiles` | `dict[str, ModelProfile]` | `light` / `strong` 模型 profile |
| `metadata_store` | `MetadataStore` | 运行记录和 saved_query 存储 |
| `target_dialect` | `str` | 目标 SQL 方言 |

节点通过 `self.dependencies.get("llm_client")` 读取。

## 配置 workflow.yaml

注册节点后，在 `workflow.yaml` 声明节点和边。

### 声明节点

```yaml
nodes:
  my_custom_node:
    type: my_custom_node              # 必须与 @register_node 的参数一致
    my_option: "some_value"           # 自定义配置，会进入 node.config
    another_option: 42
```

`type` 字段以外的所有字段都会传入节点的 `config`。

### 声明边

```yaml
edges:
  my_custom_node:
    on_success: next_node             # outcome="success" 时去 next_node
    on_failure: error_classification  # outcome="failure" 时去 error_classification
    on_my_outcome: another_node       # 自定义 outcome 路由
```

边的键名格式为 `on_<outcome>`。`terminal: true` 表示终止节点。

### 插入主链路

如果要把节点插入主链路，需要调整前驱节点的边。例如在 `sql_generation` 后插入自定义节点：

```yaml
edges:
  sql_generation:
    on_success: my_custom_node      # 原来是 on_success: sql_validation
  my_custom_node:
    on_success: sql_validation
```

## 测试模板

参考 `tests/unit/nodes/` 现有节点测试。最小测试结构：

```python
from text_to_sql_demo.workflow.state import WorkflowState
from text_to_sql_demo.workflow.dependencies import NodeDependencies
from text_to_sql_demo.nodes.my_custom import MyCustomNode


def test_my_custom_node_success():
    # 1. 构造 state
    state = WorkflowState(user_question="测试问题", request_id="req_1")
    state.data["schema_linking"] = {"tables": [...]}

    # 2. 构造节点
    node = MyCustomNode(
        name="my_custom_node",
        node_type="my_custom_node",
        config={"my_option": "test"},
        dependencies=NodeDependencies(values={"llm_client": MockLLMClient()}),
    )

    # 3. 执行
    result = node.run(state)

    # 4. 断言
    assert result.outcome == "success"
    assert result.state_patch["data"]["my_result"] == "value"
```

### 测试要求

- **不依赖真实 LLM API**：用 `MockLLMClient`（`src/text_to_sql_demo/llm/client.py`）。
- **不依赖真实数据库**：用 SQLite 或 Mock。
- **覆盖成功和失败路径**：至少测 `outcome="success"` 和一个失败 outcome。
- **集成测试**：如果节点参与主链路，补 `tests/integration/` 测试覆盖完整流转。

参考测试：
- `tests/unit/nodes/test_begin.py`：最简单的节点测试。
- `tests/unit/nodes/test_sql_generation.py`：带 LLM 依赖的节点测试。
- `tests/unit/workflow/test_engine.py`：工作流流转测试。
- `tests/unit/workflow/test_architecture_constraints.py`：架构约束测试（保护你不违反红线）。

## 架构红线

以下约束由 `tests/unit/workflow/test_architecture_constraints.py` 做 AST 检查，**违反会导致测试失败**：

1. **`WorkflowEngine` 不得导入具体节点类**。引擎只认识 `NodeFactory` 和 `NodeResult.outcome`。
2. **`NodeFactory` 不得使用 `if/elif` 或 `match/case` 分发具体节点类型**。必须通过 `NodeRegistry.get(type)` 解析。
3. **节点实现不得导入 `WorkflowEngine`**。节点不应感知工作流流转逻辑。

以下约束由代码审查和规范保证：

4. **节点流转由 `NodeResult.outcome` 和 `workflow.yaml` 的 `edges` 决定**，不要在节点内部硬编码下一节点。
5. **不要在 API handler 中构造 prompt 或写工作流流转逻辑**。这些属于 service、workflow 和 nodes。
6. **新增节点不应要求修改 `WorkflowEngine` 或 `NodeFactory`**。只需注册 + 配置 workflow.yaml。

## 现有节点参考

| 节点类型 | 实现文件 | 复杂度 | 学习建议 |
| --- | --- | --- | --- |
| `begin` | `nodes/begin.py` | 最低 | 最简单的节点，只有 20 行，适合作为起点 |
| `selection` | `nodes/selection.py` | 低 | 意图分类，无外部依赖 |
| `schema_linking` | `nodes/schema_linking.py` | 中 | 读 state.data.schema，调 SchemaLinker |
| `sql_generation` | `nodes/sql_generation.py` | 高 | 完整展示依赖注入、模型路由、prompt 构建、LLM 调用 |
| `sql_validation` | `nodes/sql_validation.py` | 中 | 展示成功/失败两种 outcome |
| `error_classification` | `nodes/error_reflection.py` | 高 | 展示多 outcome 路由（6 种策略） |
| `finalization` | `nodes/finalization.py` | 中 | 展示 terminate 节点 |

建议按从简到繁的顺序阅读源码：`begin` → `selection` → `sql_validation` → `sql_generation` → `error_classification`。
