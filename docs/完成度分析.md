# SQL 生成流程完成度分析

本文档从面试展示角度分析当前实现的完成度、亮点、限制和后续优化方向。它只描述已经落地的能力，不编造性能指标或生产级能力。

## 已实现能力

| 能力 | 当前实现 | 证据入口 |
| --- | --- | --- |
| 可配置工作流 | `workflow.yaml` 配置 nodes、edges、max steps 和 max repair attempts。 | `src/text_to_sql_demo/config/models.py` |
| 注册式节点 | 节点通过 `@register_node` 注册，`NodeFactory` 按 type 创建。 | `src/text_to_sql_demo/workflow/registry.py`、`factory.py` |
| 状态通信 | 节点通过 `WorkflowState.data` 传递 schema、examples、SQL、错误和结果。 | `src/text_to_sql_demo/workflow/state.py` |
| Schema Linking | 规则打分、中文同义词、字段描述和外键扩展。 | `src/text_to_sql_demo/schema/linking.py` |
| Top-K 示例检索 | 本地 YAML 示例，词法重叠和表重叠排序；顶层 retrieval 配置会注入到示例检索节点。 | `src/text_to_sql_demo/retrieval/examples.py` |
| 模型路由 | `ComplexityClassifier` + `ModelRouter` 在生成节点内选择 `light/strong` alias。 | `src/text_to_sql_demo/routing/complexity.py` |
| Prompt 裁剪 | 只注入 linked schema、retrieved examples 和业务方言范式，并输出 summary。 | `src/text_to_sql_demo/prompts/builder.py` |
| SQL 生成 | `GenSQLAgenticNode` 封装 prompt 构建、上下文注入、模型路由和 provider 无关 LLM 调用。 | `src/text_to_sql_demo/nodes/sql_generation.py` |
| Prompt 模板 | 生成和修复 prompt 都可以从 YAML 模板加载，默认模板位于 `configs/prompts/`。 | `src/text_to_sql_demo/prompts/templates.py` |
| SQL 方言校验 | SQLGlot parse、单语句、只读 SELECT、schema 引用、可选渲染。 | `src/text_to_sql_demo/sql/validator.py` |
| SQL 执行 | SQLAlchemy 执行已校验 SQL，节点允许 `sqlite`、`postgres`、`mysql` 三类执行方言，并要求校验方言与执行方言一致。 | `src/text_to_sql_demo/nodes/sql_execution.py`、`src/text_to_sql_demo/execution/sql_executor.py` |
| 策略反思闭环 | validation/execution 失败后由 `ReflectionDecisionNode` 路由到 FIX_SQL、RELINK_SCHEMA、REASONING_REWRITE 或 HITL，最多三轮。 | `src/text_to_sql_demo/nodes/error_reflection.py`、`sql_fix.py`、`reasoning_rewrite.py`、`hitl.py`、`src/text_to_sql_demo/reflection/` |
| Trace | engine 自动记录每个节点执行摘要。 | `src/text_to_sql_demo/workflow/engine.py` |
| 运行时配置 | 前端可创建临时数据库和 `light/strong` 模型路由配置，业务请求通过 `runtime_config_id` 使用。 | `src/text_to_sql_demo/runtime/`、`frontend/src/App.tsx` |
| 前端演示 | 查询、运行配置、SQL 编辑、结果、修复提示、开发者信息。 | `frontend/src/App.tsx` |

维护提示：如果将某项能力从“限制”推进到“已实现”，必须补测试入口，再更新本表。

## 演示亮点

1. 工作流不是硬编码在 API handler 中。`TextToSQLApiService.run_query` 只负责组装状态和调用 `WorkflowEngine.run`，节点流转由 `workflow.yaml` 的 edges 决定。
2. 节点扩展边界清楚。`WorkflowEngine` 和 `NodeFactory` 不导入具体节点类，并有架构约束测试保护。
3. Prompt 裁剪可解释。Trace 里能看到 linked schema、Top-K examples、业务方言范式和 prompt summary，方便面试时讲清楚为什么没有把完整 schema 塞给模型。
4. Mock LLM 支持稳定演示。`scripts/run_demo.py` 用 sequence/default response 覆盖成功、修复和终止路径，不依赖网络和 API Key。
5. 修复循环有明确终止条件。`attempt_count` 达到上限后走 `attempts_exhausted -> hitl -> finalization`，不会无限修复。
6. 前端把用户视角和开发者视角分开。普通区域展示 SQL 和结果，开发者信息里再展开模型路由、Few-shot、Trace 和错误。

维护提示：演示亮点必须能被 `scripts/run_demo.py` 或测试稳定复现；如果 demo 场景变化，要同步更新 [面试演示场景](面试演示场景.md)。

## 当前限制

- 默认 workflow 使用 OpenAI-compatible provider，需要正确配置 `DEEPSEEK_API_KEY` / `DEEPSEEK_BASE_URL`；测试和内置 demo 场景仍通过 `MockLLMClient` 保持确定性。
- SQL 执行路径支持 SQLite、PostgreSQL 和 MySQL 方言，但服务型数据库依赖本地安装可选驱动、只读账号和真实连接环境；当前不宣称生产级数据库权限隔离。
- Schema 来源默认是数据库 introspection；`SchemaConfig.catalog_source` 支持 `yaml` 的配置模型，但当前主链路没有 YAML schema loader。
- Example retrieval 是本地词法检索，不使用向量数据库或 embedding。
- `ComplexityClassifier` 是规则分类器，不是训练模型，也没有性能指标。
- 内存运行记录 `InMemoryRunStore` 只适合 demo，服务重启后记录消失。
- 运行时配置使用内存存储，服务重启后 `runtime_config_id` 会失效；当前不做认证、多租户或长期密钥托管。
- SQL 只读安全依赖 SQLGlot AST 和执行前校验，不等同于生产级数据库权限隔离。

维护提示：限制说明应随实现变化及时更新，尤其是 LLM provider、数据库 driver、schema loader 和检索策略。

## 后续优化方向

- 为不可修复安全错误增加明确策略，例如写入/DDL 直接 finalization，而不是进入修复循环。
- 扩展更多 LLM provider adapter，但保留 `LLMClient` 协议和 Mock 测试路径。
- 为 PostgreSQL/MySQL 真实连接场景补更完整的只读权限、连接配置和方言回归测试矩阵。
- 将 `WorkflowState.data` 中常用键逐步收敛为更细的 typed model，提升长期维护性。
- 为运行时配置面板补更细的 provider 表单校验和独立“测试连接”交互；当前是在创建配置时统一测试。
- 在前端增加更清晰的 Trace 时间线视图，但保持默认信息密度克制。

维护提示：后续优化项进入实现前，应先在相关 docs 中标注目标行为，再补单元/集成测试，最后更新 README 的能力边界。
