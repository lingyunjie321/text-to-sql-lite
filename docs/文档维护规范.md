# 文档维护规范

所有文档必须基于当前代码、配置和测试，不把规划态内容写成已实现能力。

## 基本规则

- 正文优先中文，保留必要技术术语、路径、类名和函数名。
- README 只放高层入口、常用命令、API 示例和文档导航。
- 详细架构、工作流、代码追踪、完成度和维护规范放在 `docs/`。
- 关键流程图必须先提供 boxed plaintext/ASCII 图，再提供 Mermaid 渲染版；两者都只能画真实存在的模块、节点和路径。
- 文件树只展示关键目录和职责，不展开 `node_modules`、`dist`、缓存文件或数据库二进制细节。
- 不编造性能指标、生产级能力或真实 LLM 接入状态。
- 如果文档描述一项已实现能力，应能在源码或测试中找到证据。

维护提示：每次改文档前，先用 `rg` 查真实符号和路径；不要从旧规划文档复制未验证内容。

## 改 workflow 时

触发条件：

- 修改 `workflow.yaml` 的 `nodes`、`edges`、`workflow.max_steps`、`workflow.max_repair_attempts`。
- 新增或重命名 outcome，例如 `validation_success`、`fix_sql`、`reasoning_rewrite`。
- 调整模型 alias、database 配置或 dialect 配置。

需要同步更新：

- `README.md` 的核心能力、快速启动或 API 示例。
- `docs/文本转SQL工作流.md` 的节点表、plaintext/Mermaid 图、成功/修复/终止路径。
- `docs/整体架构.md` 的 plaintext/Mermaid 架构图和配置边界。
- `docs/面试演示场景.md` 的演示流程。
- `docs/SQL生成过程代码追踪.md` 的调用链。

建议验证：

```bash
ruff check .
python -m pytest tests/unit/test_config_loader.py tests/unit/workflow tests/integration/test_demo_scenarios.py
```

维护提示：workflow 流转必须由 `NodeResult.outcome` 和配置决定，文档中不要写成 API handler 硬编码。

## 改 nodes 时

触发条件：

- 新增节点类或修改现有节点输出。
- 改 `state.data` key，例如 `schema_linking`、`retrieved_examples`、`prompt_summary`、`repair_history`。
- 修改节点注册 type，例如 `reflection_decision`、`reflection_fix`、`error_classification`。

需要同步更新：

- `docs/文本转SQL工作流.md` 节点职责表和 plaintext/Mermaid 图。
- `docs/SQL生成过程代码追踪.md` 对应步骤。
- `docs/项目结构与模块职责.md` 模块职责表。
- `docs/完成度分析.md` 已实现能力或限制。

建议验证：

```bash
ruff check .
python -m pytest tests/unit/workflow tests/unit/nodes tests/integration/test_sql_repair_workflow.py
```

维护提示：节点实现不得导入 `WorkflowEngine`；如果为了实现需求想这么做，先回到架构设计而不是只改文档。

## 改 API 时

触发条件：

- 修改 `src/text_to_sql_demo/main.py` 的路由路径或响应。
- 修改 `src/text_to_sql_demo/api/models.py` 的请求模型。
- 修改 `serialize_run` 的 payload 字段。

需要同步更新：

- `README.md` 的 API 示例。
- `docs/SQL生成过程代码追踪.md` 的入口和 finalization 响应。
- `docs/整体架构.md` 的 API 层职责。
- `frontend/src/api/types.ts` 和 `frontend/src/api/client.ts`，如果前端依赖该字段。

建议验证：

```bash
ruff check .
python -m pytest tests/integration/test_api_workflow.py
cd frontend && npm test
```

维护提示：API handler 不应构造 prompt，也不应写 workflow 流转逻辑；这些属于 service、workflow 和 nodes。

## 改 prompt、模型路由或 LLM 时

触发条件：

- 修改 `PromptBuilder.build`。
- 修改 `ComplexityClassifier` 或 `ModelRouter`。
- 修改 `LLMClient`、`MockLLMClient` 或新增 provider adapter。
- 改模型 alias，例如 `light`、`strong`。
- 改 `workflow.yaml` 中 provider、模型名、`api_key_env` 或 `base_url_env`。

需要同步更新：

- `README.md` 的核心能力、默认 LLM 说明和 `.env.example`。
- `docs/SQL生成过程代码追踪.md` 的第 6、7 节。
- `docs/完成度分析.md` 的亮点、限制和后续方向。
- `docs/面试演示场景.md` 的模型路由讲解。

建议验证：

```bash
ruff check .
python -m pytest tests/unit/prompts tests/unit/routing tests/unit/llm tests/integration/test_demo_scenarios.py
```

维护提示：不要在业务代码中硬编码真实模型名称；文档也应使用模型 alias 描述业务选择。

## 改 schema、examples 或数据库时

触发条件：

- 修改 demo schema 或种子数据。
- 修改 `SchemaLinker` 的匹配规则。
- 修改 `configs/examples.yaml` 或示例检索策略。
- 修改数据库 URL 或初始化脚本。

需要同步更新：

- `README.md` 初始化数据库和 Demo 场景。
- `docs/项目结构与模块职责.md` 配置与数据文件。
- `docs/SQL生成过程代码追踪.md` 的 schema linking 和 retrieval 步骤。
- `docs/面试演示场景.md` 的 Scenario SQL 和讲解点。

建议验证：

```bash
ruff check .
python -m pytest tests/unit/schema tests/unit/retrieval tests/integration/test_database_initialization.py tests/integration/test_demo_scenarios.py
python scripts/run_demo.py
```

维护提示：如果字段名或表关系变化，修复路径场景很可能需要一起调整。

## 改前端交互时

触发条件：

- 修改 `frontend/src/App.tsx` 的主要交互。
- 修改 `frontend/src/api/types.ts`、`api/client.ts` 或 `adapters/queryRunAdapter.ts`。
- 修改 SQL 编辑、修复提示、结果表格、开发者 Trace 展示。

需要同步更新：

- `README.md` 前端启动说明。
- `docs/整体架构.md` 前端层说明。
- `docs/面试演示场景.md` 面试讲解顺序。
- `docs/完成度分析.md` 前端亮点或限制。

建议验证：

```bash
cd frontend && npm test
cd frontend && npm run build
```

维护提示：前端展示的是 demo 产品体验，不要在文档中把它描述成带认证、多租户或生产运维能力的平台。

## 文档维护提示集中规则

以下规则原散落在各业务文档小节末尾，现统一收敛到本规范。业务文档末尾只保留一句"维护规范见 [文档维护规范](文档维护规范.md)"。

- **改 workflow 时**：同步更新 README 核心能力/快速启动/API 示例、文本转SQL工作流文档的节点表和流转图、整体架构的架构图和配置边界、面试演示场景、SQL生成过程代码追踪。验证命令见上方"改 workflow 时"小节。
- **改 nodes 时**：同步更新文本转SQL工作流文档节点职责表和流转图、SQL生成过程代码追踪对应步骤、项目结构与模块职责模块职责表、完成度分析已实现能力或限制。
- **改 API 时**：同步更新 README API 示例、SQL生成过程代码追踪入口和 finalization 响应、整体架构 API 层职责、前端 `api/types.ts` 和 `api/client.ts`。
- **改 prompt/模型路由/LLM 时**：同步更新 README 核心能力和默认 LLM 说明和 `.env.example`、SQL生成过程代码追踪第 6/7 节、完成度分析亮点/限制/后续方向、面试演示场景模型路由讲解。
- **改 schema/examples/数据库时**：同步更新 README 初始化数据库和 Demo 场景、项目结构与模块职责配置与数据文件、SQL生成过程代码追踪 schema linking 和 retrieval 步骤、面试演示场景 Scenario SQL 和讲解点。
- **改前端交互时**：同步更新 README 前端启动说明、整体架构前端层说明、面试演示场景面试讲解顺序、完成度分析前端亮点或限制。
- **新增 API 入口/节点目录/数据源/LLM adapter/前端调用方式**：同步更新整体架构的 plaintext 和 Mermaid 架构图和模块职责表，不要把尚未实现的模块画进架构图。
- **新增 `state.data` 键/改变 API 序列化字段/调整 trace 摘要**：同步更新 SQL生成过程代码追踪、文本转SQL工作流文档、前端 `frontend/src/api/types.ts`。
- **新增数据库 driver**：不要只改 executor；还要更新配置说明、driver 到 SQLGlot dialect 映射、只读保障策略、测试矩阵和 README 限制。
- **调整 `max_attempts` 上限/`max_repair_attempts` 默认值/不可修复错误策略**：同步更新终止路径说明和终止路径测试。
- **将能力从"限制"推进到"已实现"**：必须先补测试入口，再更新完成度分析表格。

维护提示：workflow 流转必须由 `NodeResult.outcome` 和配置决定，文档中不要写成 API handler 硬编码。节点实现不得导入 `WorkflowEngine`；如果为了实现需求想这么做，先回到架构设计而不是只改文档。

## 发布前文档核对清单

- [ ] README 的后端启动命令包含 `PYTHONPATH=src` 或说明 editable install。
- [ ] README 的 API 示例只引用当前存在的接口。
- [ ] plaintext 图和 Mermaid 图中的节点名都和 `workflow.yaml` 一致。
- [ ] docs 中没有不存在的旧规划目录。
- [ ] docs 中没有未实现的旧节点名。
- [ ] 文件树没有展开 `node_modules`、`dist`、缓存目录。
- [ ] 默认 LLM 说明与 `workflow.yaml` 一致；如果默认 provider 需要 API Key，README 和 `.env.example` 必须写清楚。测试和 `scripts/run_demo.py` 使用 `MockLLMClient` 的边界也要写清楚。
- [ ] 运行 `ruff check .`、`python -m pytest`、`python scripts/run_demo.py`。
- [ ] 如果改到前端说明，运行 `cd frontend && npm test` 和 `cd frontend && npm run build`。
