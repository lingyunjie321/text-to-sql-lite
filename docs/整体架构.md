# 整体架构

本文档描述当前仓库真实实现，不包含规划态目录或未落地模块。

项目目标是展示一个清晰、可测试、可配置的 Text-to-SQL Agent 主链路：API 接收自然语言问题，工作流节点按配置流转，默认 workflow 已对齐 Datus-style 的 `Begin -> Selection -> Schema -> Context Retrieval -> GenSQL -> Execute` 核心链路，同时保留本项目现有 LLM alias 抽象，并提供轻量策略反思闭环与 SQLContext 工作记忆。

默认 workflow 通过 OpenAI-compatible provider 调用 `light` / `strong` 模型 alias，SQLGlot 做方言和只读校验，SQLAlchemy 执行已校验 SQL，只读结果和 Trace 返回给前端或 API 调用方。

测试和 `scripts/run_demo.py` 会显式注入 `MockLLMClient`，不依赖真实 LLM API。

## 分层职责

| 层次 | 真实路径 | 当前职责 |
| --- | --- | --- |
| 前端演示 | `frontend/src/` | 提交自然语言问题、展示 SQL/结果/修复提示/开发者 Trace，通过 Vite proxy 调用 `/api`。 |
| API 层 | `src/text_to_sql_demo/main.py`、`src/text_to_sql_demo/api/models.py` | 定义 FastAPI app、请求模型、统一错误响应和现有接口。 |
| 应用服务 | `src/text_to_sql_demo/api/service.py` | 加载配置、初始化数据库、读取 schema、创建 `WorkflowEngine`、序列化运行结果和内存运行记录。 |
| 运行时配置 | `src/text_to_sql_demo/runtime/` | 管理前端创建的临时数据库和 `light/strong` 模型路由配置，解析为请求级 workflow 依赖。 |
| 配置层 | `workflow.yaml`、`src/text_to_sql_demo/config/` | 用 Pydantic 校验 workflow、nodes、edges、database、models、dialect、trace 等配置引用。 |
| 工作流核心 | `src/text_to_sql_demo/workflow/` | 提供 `WorkflowState`、`NodeResult`、`BaseNode`、`NodeRegistry`、`NodeFactory`、`WorkflowEngine`。 |
| 业务节点 | `src/text_to_sql_demo/nodes/` | 实现 begin、selection、schema linking、context retrieval、example retrieval、SQL generation、validation、execution、reflection/fix、reasoning rewrite、HITL、finalization。 |
| 策略反思 | `src/text_to_sql_demo/reflection/` | 定义反思策略、决策模型、SQLAttemptContext 和 SQLContext 脱敏摘要格式化。 |
| LLM 抽象 | `src/text_to_sql_demo/llm/` | 定义 provider 无关协议、`MockLLMClient`、OpenAI-compatible adapter 和 alias 路由 client；默认 workflow 由配置决定 provider，测试不依赖真实付费 API。 |
| Prompt 与路由 | `src/text_to_sql_demo/prompts/`、`src/text_to_sql_demo/routing/` | 只用 linked schema、Top-K examples、RAG 上下文和业务范式构造 prompt，根据复杂度选择模型 alias。 |
| Schema 与检索 | `src/text_to_sql_demo/schema/`、`src/text_to_sql_demo/retrieval/` | 从数据库 introspection 读取 schema，本地规则 linking，本地 YAML 示例和知识库 Top-K 词法检索。 |
| SQL 校验执行 | `src/text_to_sql_demo/sql/`、`src/text_to_sql_demo/execution/` | SQLGlot 方言解析/转换、只读 SELECT 和 schema 校验，SQLAlchemy 执行 SQLite/PostgreSQL/MySQL 等已配置只读连接。 |
| 测试 | `tests/`、`frontend/src/**/*.test.tsx` | 覆盖架构约束、节点、SQL 校验、API、修复闭环、demo 场景和前端交互。 |

## 工作流核心设计

`WorkflowEngine.run` 只认识配置、`NodeFactory` 和 `NodeResult.outcome`。它不会导入具体节点类，也不会按节点类型写 `if/elif` 分支；这些约束由 `tests/unit/workflow/test_architecture_constraints.py` 做 AST 检查。

节点创建流程是：

1. `TextToSQLApiService.__init__` 导入 `text_to_sql_demo.nodes`，触发各节点上的 `@register_node(...)` 装饰器。
2. `NodeFactory.create` 从 `NodeRegistry` 按 `NodeConfig.type` 取节点类。
3. `WorkflowEngine.run` 执行节点，合并 `NodeResult.state_patch` 到 `WorkflowState`。
4. `WorkflowEngine._resolve_next_node` 根据当前节点的 outcome 和 `workflow.yaml` 的 `edges` 决定下一节点。

## 状态与 Trace

`WorkflowState` 是 typed Pydantic model，但业务中间结果集中放在 `state.data`：

- 输入和配置快照：`schema`、`target_dialect`、`runtime_config_id`、`max_repair_attempts`、`debug`。
- 任务与意图：`task`、`intent`。
- 生成上下文：`schema_linking`、`rag_context`、`retrieved_examples`、`available_example_count`、`prompt_summary`。
- SQL 状态：`generated_sql`、`current_sql`、`validated_sql`、`validation_result`。
- 执行与反思修复：`execution_result`、`last_error`、`reflection_decision`、`sql_contexts`、`last_reflection_strategy`、`repair_instruction`、`repair_instruction.strategy`、`repair_history`、`attempt_count`、`hitl_reason`。
- 最终输出：`final_status`、`final_sql`、`final_result`、`final_error`、`termination_reason`。

每个节点执行后，`WorkflowEngine._execute_node` 追加 `TraceEvent`，包含 `node_name`、`node_type`、`status`、`outcome`、`duration_ms`、`input_summary`、`output_summary` 和错误摘要。Trace 只记录摘要，避免把完整 prompt 或凭据放进响应。

## 数据与外部依赖边界

**默认数据库**来自 `workflow.yaml` 的 `database.connections.demo_sqlite.fallback_url`：`sqlite:///data/sqlite/demo.db`。如果设置了 `DEMO_DATABASE_URL`，`TextToSQLApiService` 会优先使用环境变量。

**SQLite 自动发现**：`database.sqlite_discovery` 默认扫描 `data/sqlite/` 下的可见 `.db` 文件，并把未在 workflow 中显式配置的文件作为只读数据库预设返回给前端；`metadata.db` 默认排除，因为它属于项目内部运行记录库。SQLite 数据由 `scripts/init_db.py` 或 `TextToSQLApiService.ensure_database` 初始化。

**服务型数据库**：PostgreSQL/MySQL 连接通过结构化配置或 URL 环境变量解析。Schema 统一通过 SQLAlchemy inspector 读取。

**默认知识库**来自 `configs/knowledge.yaml`，包含 Reference SQL、文档片段、Metric 和 Semantic Model 的本地 YAML fallback。当前实现使用词法 Top-K 与 linked tables 加权；如果后续接入 LanceDB/FastEmbed/PyArrow，应保持 `KnowledgeStore` 或等价接口可替换，并保留不依赖向量服务的测试路径。

**默认 LLM**：`workflow.yaml` 默认把 `light`、`strong` 配置为 `openai_compatible` provider，并从 `DEEPSEEK_API_KEY` / `DEEPSEEK_BASE_URL` 读取真实调用配置。业务代码只依赖 alias，不直接绑定供应商模型名；测试、集成场景和 `scripts/run_demo.py` 通过依赖注入使用 `MockLLMClient`。
