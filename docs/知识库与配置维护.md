# 知识库与配置维护

本文档面向**数据团队**，说明如何维护 Text-to-SQL Agent 的可信上下文。项目核心定位是"由数据团队维护可信上下文"，三个 YAML 文件和 saved_query 治理是这条上下文链路的主体。判断当前字段和检索逻辑时，以源码为准。

## 上下文链路全景

一次 Text-to-SQL 请求会把以下可信上下文裁剪后注入 prompt：

```text
用户问题
   │
   ▼
schema_linking 选出相关表列
   │
   ├──► configs/examples.yaml        ── 本地 SQL 示例（Few-shot）
   ├──► configs/knowledge.yaml       ── Reference SQL / 文档 / Metric / Semantic Model
   ├──► configs/dialect_patterns.yaml ── 业务方言范式
   └──► approved saved_query         ── 运行沉淀的可信 Reference SQL
                  │
                  ▼
            PromptBuilder 裁剪后注入 prompt
```

四类上下文都走词法 Top-K 检索，不使用向量数据库。`draft` / `deprecated` 状态的 saved_query **不会**进入 prompt。

## configs/examples.yaml：本地 SQL 示例

`ExampleRetrievalNode` 读取此文件，作为 Few-shot 示例注入 prompt。

### 字段说明

```yaml
examples:
  - natural_language: 统计每个地区订单金额      # 必填，自然语言问题
    sql: |                                       # 必填，对应 SQL
      SELECT r.name AS region_name, SUM(o.amount) AS total_order_amount
      FROM regions r
      JOIN customers c ON c.region_id = r.id
      JOIN orders o ON o.customer_id = c.id
      GROUP BY r.name
    dialect: sqlite                              # 必填，SQL 方言，会按 dialect 过滤
    tags: ["地区", "订单", "金额", "聚合"]         # 选填，参与词法匹配
    involved_tables: ["regions", "customers", "orders"]  # 必填，参与表重叠加权
```

### 检索权重

`ExampleStore.search`（`src/text_to_sql_demo/retrieval/examples.py`）打分逻辑：

| 命中类型 | 权重 |
| --- | --- |
| 词项匹配（query token ∩ example 的 natural_language + tags + involved_tables token） | 每个匹配词项 +2.0 |
| 表重叠（linked schema 表 ∩ example.involved_tables） | 每张表 +1.5 |
| 问题完全匹配 query == example.natural_language | 额外 +5.0 |

`dialect` 不匹配的示例直接跳过。只有 score > 0 的示例才进入 Top-K 结果。

### 维护建议

- **改 schema 后必须同步**：表名、字段名变化时，相关示例的 `involved_tables` 和 `sql` 要一起改，否则检索会失配。
- **tags 用中文业务词**：如"地区""金额""排名"，提高词法命中率。
- **每个业务场景至少 1 条**：聚合、排名、多表关联、时间分桶等高频场景都应有示例。
- **dialect 要写对**：写错 dialect 的示例永远不会被检索到。

## configs/knowledge.yaml：业务知识库

`ContextRetrievalNode` 读取此文件，作为 Reference SQL / 文档 / Metric / Semantic Model 注入 prompt。

### 四类知识字段

**Reference SQL**（`reference_sql`）：可信的成熟 SQL，作为生成参考。

```yaml
reference_sql:
  - name: order_amount_detail              # 必填，唯一标识
    natural_language: 列出订单金额          # 必填，用于词法匹配
    sql: SELECT id, amount FROM orders ORDER BY id  # 必填
    involved_tables: [orders]              # 选填，参与表重叠
    tags: [订单, 金额]                      # 选填，参与词法匹配
```

**Documents**（`documents`）：业务口径、字段说明等文本知识。

```yaml
documents:
  - title: 订单金额口径                    # 必填
    content: 订单金额默认使用 orders.amount 字段，按订单粒度统计。  # 必填
    tags: [订单, 金额]                     # 选填
```

**Metrics**（`metrics`）：指标定义，帮助模型理解业务指标口径。

```yaml
metrics:
  - name: total_order_amount              # 必填
    description: 订单总金额，统计口径为订单表 amount 字段求和。  # 必填
    expression: SUM(orders.amount)        # 选填，指标表达式
    involved_tables: [orders]             # 选填
```

**Semantic Models**（`semantic_models`）：语义层说明，描述表和字段的业务含义。

```yaml
semantic_models:
  - name: orders_semantic                 # 必填
    description: 订单事实表，包含订单金额、订单日期、客户和状态。  # 必填
    tables: [orders]                      # 选填
```

### 检索逻辑

`KnowledgeStore.search`（`src/text_to_sql_demo/retrieval/knowledge.py`）对四类知识分别做词法 Top-K：

- 按 query 词项与 item 的 natural_language / tags / title / content / name / description 词法重叠打分。
- Reference SQL 和 Metric 额外按 linked tables 与 `involved_tables` 重叠加权。
- 四类各自返回 Top-K，合并写入 `state.data.rag_context`。

### 维护建议

- **Reference SQL 和 examples 的区别**：examples 是 Few-shot（直接教模型写 SQL），Reference SQL 是参考（告诉模型"这个指标应该这么算"）。同一个业务问题可以两边都放，但侧重点不同。
- **Documents 放口径**：如"金额用 amount 不用 total_amount""地区按 region_id 关联"等字段口径，避免模型用错字段。
- **Metrics 放定义**：指标的统计口径、时间窗口、过滤条件，而不是具体 SQL。
- **Semantic Models 放语义**：表的业务含义、粒度说明，帮助模型理解表用途。

## configs/dialect_patterns.yaml：业务方言范式

`GenSQLAgenticNode` 读取此文件，按问题、方言和 linked tables 检索业务 SQL 范式，注入 prompt。

### 字段说明

```yaml
patterns:
  - name: sqlite_region_rank_window        # 必填，唯一标识
    dialect: sqlite                        # 必填，会按 dialect 过滤
    description: 地区内 Top-N 排名优先使用窗口函数  # 必填，说明范式用途
    pattern: |                             # 必填，SQL 范式模板
      SELECT *
      FROM (
        SELECT region, customer_name, total_amount,
               RANK() OVER (PARTITION BY region ORDER BY total_amount DESC) AS region_rank
        FROM region_customer_totals
      ) ranked
      WHERE region_rank <= 3
    tags: ["地区", "排名", "Top-N", "窗口函数"]  # 选填，参与词法匹配
    involved_tables: ["regions", "customers", "orders"]  # 选填，参与表重叠
```

### 检索逻辑

`PatternStore.search`（`src/text_to_sql_demo/retrieval/patterns.py`）：

- 按 `dialect` 过滤。
- 按 query 词项与 pattern 的 description + tags 词法重叠打分。
- 按 linked tables 与 `involved_tables` 重叠加权。
- 返回 Top-K（默认 `patterns_top_k: 3`，在 `workflow.yaml` 的 `sql_generation` 节点配置）。

### 维护建议

- **放跨场景通用的范式**：如"Top-N 用窗口函数""按月分桶用 DATE_TRUNC"，而不是具体某条 SQL。
- **description 要写清楚什么时候用**：如"地区内 Top-N 排名优先使用窗口函数，先聚合再排名"。
- **方言要写对**：SQLite 和 PostgreSQL 的窗口函数、日期函数写法不同，范式要分方言维护。

## saved_query 治理流

saved_query 是运行沉淀的可复用 SQL，只有 `approved` 状态才进入 prompt 检索。这是跨运行知识积累的主路径。

### 状态流转

```text
POST /api/v1/saved-queries          PATCH /api/v1/saved-queries/{id}/status
       │                                    │
       ▼                                    ▼
   ┌─────────┐   数据团队审核通过    ┌──────────┐   不再使用    ┌────────────┐
   │  draft  │ ──────────────────► │ approved │ ───────────► │ deprecated │
   └─────────┘                      └──────────┘              └────────────┘
       │                                 │                          │
       │           不进入 prompt          │ 进入 prompt 检索           │ 不进入 prompt
       └─────────────────────────────────┴──────────────────────────┘
```

### 创建 saved_query

```bash
curl -X POST http://127.0.0.1:8000/api/v1/saved-queries \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "地区订单金额统计",
    "request_id": "可选，关联某次运行",
    "question": "统计每个地区的订单总金额",
    "sql": "SELECT r.name, SUM(o.amount) FROM regions r JOIN customers c ON c.region_id=r.id JOIN orders o ON o.customer_id=c.id GROUP BY r.name",
    "tags": ["地区", "订单", "金额"],
    "status": "draft"
  }'
```

默认创建为 `draft`，不进入 prompt。

### 审核为 approved

数据团队确认 SQL 口径正确后，更新状态：

```bash
curl -X PATCH http://127.0.0.1:8000/api/v1/saved-queries/{saved_query_id}/status \
  -H 'Content-Type: application/json' \
  -d '{"status": "approved"}'
```

`approved` 后，该 saved_query 会被 `search_approved_saved_query_reference_sql`（`src/text_to_sql_demo/memory/trusted.py`）检索到，与 YAML Reference SQL 合并去重后注入 prompt。

### approved 进入 prompt 的机制

`ContextRetrievalNode` 在检索时会：

1. 从 YAML knowledge 加载 Reference SQL。
2. 从 metadata store 加载 approved saved_query，转为 `ReferenceSqlItem`。
3. 两路结果按 name 和 sql 去重，合并 Top-K 后写入 `rag_context.reference_sql`。

### 退出为 deprecated

不再使用的 approved SQL 可更新为 `deprecated`，退出检索但不删除记录（保留治理痕迹）。

### 治理边界

- 当前没有权限控制：任何调用方都能改状态，真实权限留给后续产品化。
- 当前没有审核 UI：通过 PATCH 接口手动审核。
- 当前没有批量治理：逐条操作。
- `SUCCESS` SQLContext（单次运行成功记忆）**不会**自动升级为 approved，必须人工审核。

## 与 schema 的联动

改 `scripts/init_db.py` 的 demo schema 或连接新数据库时，必须同步检查：

| 改动 | examples.yaml | knowledge.yaml | dialect_patterns.yaml |
| --- | --- | --- | --- |
| 表名变化 | 改 `involved_tables` 和 `sql` | 改 `involved_tables` | 改 `involved_tables` |
| 字段名变化 | 改 `sql` | 改 `documents.content` 口径说明 | 改 `pattern` |
| 新增表 | 可补示例 | 可补 Semantic Model | 按需补范式 |
| 新增方言 | 补对应 dialect 示例 | 按需补 Metric | 补对应 dialect 范式 |

维护规范见 [文档维护规范](文档维护规范.md)。
